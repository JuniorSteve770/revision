## PARTIE 1 — SLIDES DE RÉVISION

### Style attendu
Chaque slide suit cette structure fixe :

**[Nom du concept]**
- **Définition** : [1-2 phrases claires, concrètes, sans jargon inutile]
- **Caractéristiques** : `mot-clé-1`, `mot-clé-2`, `mot-clé-3`
- **En trading** : [exemple concret ancré dans le contexte salle de marché / desk de trading / système d'ordres]

### Règles de style
- Définitions courtes et professionnelles, pas de paraphrase
- Les mots-clés C# sont toujours en `code inline`
- Les séparateurs ◆ indiquent un changement de point — chaque ◆ commence sur une nouvelle ligne
- Les exemples trading doivent être précis : nommer les classes (Order, Trade, Derivative, PricingEngine, RiskStrategy…), les méthodes (Execute(), Calculate(), PlaceOrder()…) et les contextes (Bloomberg feed, VaR, P&L, desk Rates…)

### Exemples de slides attendus

Encapsulation :
Définition : regrouper les données et comportements dans une classe tout en contrôlant l'accès via des getters/setters.
Caractéristiques : `private`, `public`, `get`, `set`.
En trading : `Order._price` est `private`. Seule la méthode `Execute()` peut modifier le statut — aucun code externe ne peut corrompre un ordre en cours.

SOLID — DIP :
Définition : dépendre d'abstractions plutôt que d'implémentations concrètes.
Caractéristiques : `interface`, injection par constructeur, `AddTransient`/`AddScoped`/`AddSingleton`.
En trading : le moteur de risk reçoit `IMarketDataService` — jamais `new MarketDataService()`. Changer de source Bloomberg → Reuters ne touche pas au domaine.

## PARTIE 2 — QCM PROGRESSIF

### Structure des niveaux
Produis exactement trois niveaux de difficulté avec le nombre de questions indiqué :

**Niveau Moyen** [N questions]
Objectif : vérifier la compréhension des définitions, mots-clés et comportements de base.
Types de questions à inclure :
- Définition → concept ("À quoi sert `virtual` ?")
- Concept → mots-clés ("Quels mots-clés caractérisent l'Encapsulation ?")
- Mots-clés → concept ("Les mots-clés `virtual` et `override` appartiennent à quel pilier POO ?")
- Correspondance ("Associez : `private` → ? / `override` → ? / `:` → ?")
- Principe SOLID simple ("Quel principe SOLID est violé si… ?")

**Niveau Avancé** [N questions]
Objectif : tester la capacité à identifier des patterns et principes dans du code réel, et à relier les concepts entre eux.
Types de questions à inclure :
- Lecture de code → identification de pattern ("Vous voyez `private static Lazy<T>`. Quel pattern ?")
- Lecture de code → identification multi-concepts ("Identifiez les concepts POO et principes SOLID présents dans ce code")
- POO ↔ Pattern ("Quel concept POO est au cœur du pattern Singleton ? Pourquoi ?")
- POO ↔ SOLID ("Le principe SRP est la version architecturale de quel pilier POO ?")
- SOLID ↔ Pattern ("Le pattern Repository s'appuie sur quel principe SOLID pour permettre les tests ?")
- ACID ↔ Pattern ("Quel pattern garantit qu'un seul `SaveChanges()` est appelé pour plusieurs Repositories ?")
- Anti-patterns ("Lequel de ces codes viole l'Encapsulation / le LSP / le DIP ?")

**Niveau Expert** [N questions]
Objectif : évaluer la maîtrise architecturale, la capacité à lire et diagnostiquer du code complexe, et à raisonner comme en entretien technique senior.
Types de questions à inclure :
- Toutes les combinaisons du niveau Avancé, avec plus de concepts croisés simultanément
- ACID ↔ SOLID ("Quel principe SOLID garantit que la logique de transaction n'est pas dans la classe métier ?")
- ACID ↔ POO ("L'Atomicité dans ACID correspond à quel concept POO ? Justifiez.")
- Refactoring ("Ce code viole SRP. Quelle est la bonne refactorisation parmi ces options ?")
- Prérequis entre concepts ("OCP est inapplicable sans quel mécanisme POO fondamental ?")
- Erreurs de conception en trading ("Un dev enregistre un PricingContext en AddSingleton alors qu'il contient l'état du trader connecté. Quel concept POO a-t-il ignoré ?")
- Analyse complète d'un bloc de code ("Listez tous les concepts POO, principes SOLID et patterns présents dans ce code")
- Nommage inversé ("Une classe avec constructeur `private`, propriété `static Instance`, et `Lazy<T>` s'appelle ?")

### Format de chaque question
Chaque question doit suivre ce format JSON :

{
  "question": "Question claire et directe, avec `code inline` si pertinent",
  "options": [
    "Option A — plausible mais incorrecte",
    "Option B — correcte",
    "Option C — plausible mais incorrecte",
    "Option D — piège technique réaliste"
  ],
  "answer": "Option B — correcte",
  "explanation": "Explication pédagogique courte : la règle, le raisonnement, et l'exemple trading concret qui illustre pourquoi c'est correct."
}

### Règles de qualité pour les questions

Options incorrectes :
- Toujours plausibles — jamais absurdes
- Au moins une option doit être un piège technique réaliste (ex : confondre `new` et `override`, confondre `AddScoped` et `AddSingleton`)
- Les options ne doivent pas se contredire au point de rendre la bonne réponse évidente

Explications :
- Commencer par la règle ou le concept central
- Inclure un contre-exemple ou le "pourquoi ça casse" si c'est un piège
- Terminer par un ancrage trading concret si possible
- Jamais plus de 4 phrases

Progression :
- Les questions du QCM doivent suivre la logique du cours : les concepts vus en premier sont testés en premier
- Ne pas tester un concept avancé avant que le concept prérequis ait été testé

### Combinaisons prioritaires à toujours inclure

Dans chaque QCM généré, s'assurer qu'il y a au moins une question de chacune de ces combinaisons :

| Combinaison | Exemple |
|---|---|
| Mots-clés → Concept POO | "`virtual` + `override` = ?" |
| Concept POO → Pattern | "Singleton repose sur quel concept POO ?" |
| POO ↔ SOLID | "SRP est la version architecturale de quel pilier POO ?" |
| SOLID ↔ Pattern | "Repository + DIP" |
| Code → Multi-concepts | Lire un bloc et identifier tout ce qui s'y trouve |
| ACID ↔ Pattern | "Unit of Work + Atomicité" |
| ACID ↔ SOLID | "SRP + séparation de la logique transactionnelle" |
| Anti-pattern | "Ce code viole X — pourquoi ?" |
| Prérequis entre concepts | "OCP est impossible sans quel mécanisme POO ?" |
| Nommage inversé | Donner les propriétés, trouver le nom |
